乱码根因

`ths_jq_fast_v5_deep_analysis.md/html` 的乱码不是浏览器编码问题，而是文件内容已经被写成了 `??`。根因是上次生成报告时，通过 PowerShell 管道把含中文的 Python 脚本传给解释器，中文在进入 Python 前被当前系统代码页替换成了问号。以后生成中文报告应避免这条链路：优先使用 UTF-8 文件、从已有 UTF-8 Markdown 生成 HTML，或用 ASCII 标题加 UTF-8 正文。

同目录的 `wufu_v5_external_weak_iteration.md` 能正常包含中文，说明浏览器和 UTF-8 本身没有问题。

V6 定位

V5 弱市注入版只能作为同步验证版，不能作为长期自动化终版。V6 在 V5 基础上继续推进三件事：

  • 只针对 V5 剩余的 24 个目标不一致日期输出评分明细。
  • 同时输出这 24 天的成交额阈值和过滤后池子明细。
  • 执行层统一为保留 0.2% 现金、按 100 份向下取整、先卖后买、失败订单只记录不追单，并输出日终持仓 `WUFU_POSITION`。

平台脚本输出

  • 同花顺 V6 脚本：`reports/ths_wufu_fast_v6_diagnostics_execution.py`
  • 聚宽 V6 脚本：`reports/jq_wufu_fixed_pool_v6_diagnostics_execution.py`

合理解释

V5 已经证明主矛盾是 `000510` 数据源导致的弱市状态分叉。V6 之后，剩余差异应主要来自：

1. 13:10 当前价差异：同花顺和聚宽在分钟 bar、最新价、复权/未复权处理上可能不完全一致。

2. 当日成交量差异：`today_volume` 字段口径可能不同，会影响放量过滤。

3. 池子边界差异：V5 剩余差异中多次出现同花顺池子比聚宽少 2 到 3 个，说明成交额字段或阈值边界仍有轻微偏差。

4. Top1 分数接近：很多差异日 Top10 交集达到 9/10 或 10/10，说明不是大逻辑错误，而是候选排名前几名互换。

5. 执行层差异：信号可以接近一致，但净值还会被现金不足、100 份取整、撮合价格和订单失败处理影响。

下一轮验证方法

1. 用 V6 同花顺和聚宽脚本重新跑分钟级日志。

2. 检查 24 个日期是否都出现 `WUFU_SCORE_DETAIL / WUFU_POOL_FILTER / WUFU_THRESHOLD_DETAIL`。

3. 对比两平台 Top10 逐项分数，先定位 Top1 互换是价格、成交量、R2 还是池子过滤导致。

4. 检查 `WUFU_POSITION`，确认两平台日终持仓数量是否一致。

5. 如果信号差异仍集中在池差日期，下一版优先统一成交额字段和阈值四舍五入口径。

6. 如果信号已完全一致但净值仍分化，下一版继续推进执行撮合和订单失败处理。

长期方案

V5/V6 的弱市注入是验证版。长期自动化仍建议继续查找同花顺可用的中证 A500 指数源；如果平台内确实没有可用源，再考虑本地每日生成弱市区间或 CSV，再同步给同花顺脚本。
