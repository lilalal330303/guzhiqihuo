# 同花顺快速 V3 vs 聚宽固定池 V3 分钟日志对比报告

## 结论摘要

本轮 V3 已经把“成交额阈值口径”基本对齐，但“弱市状态机”仍未对齐。

- 共同交易日：`1575` 天，区间 `2020-01-02` 至 `2026-07-06`。
- 目标 ETF 匹配：`1427 / 1575`，匹配率 `90.60%`。
- Top10 平均重合：`5.60 / 10`。
- 弱市状态匹配：`1415 / 1575`，匹配率 `89.84%`，仍有 `160` 天不一致。
- 成交额阈值 ratio 中位数：`0.9932`，均值 `0.9930`，已经非常接近 1。
- 同花顺收盘资金：`99.85 万` 到 `5156.76 万`，按首日收盘估算收益 `5064.54%`，最大回撤 `-22.23%`。

和 V2 相比，V3 的主要进展不是目标匹配率大幅提升，而是把阈值差异从 `0.37` 修到 `0.99`，证明固定池阈值口径已经对齐。

## 速度与日志体积

V3 的日志瘦身有效：

- 同花顺日志行数：从 V2 的 `8929` 行降到 V3 的 `7397` 行。
- 聚宽日志行数：从 V2 的 `10079` 行降到 V3 的 `5393` 行。
- 同花顺日志大小：约 `1.11MB`。
- 聚宽压缩包大小：约 `120KB`。

V3 默认关闭动态池、全市场阈值、缓存日志和评分明细，仅保留核心信号和少量弱市边界日志。速度方向是正确的。

## 核心拆解

弱市状态仍然是第一主因：

- 弱市一致的 `1415` 天里，目标匹配率 `98.37%`。
- 弱市不一致的 `160` 天里，目标匹配率只有 `21.88%`。

所以当前两平台的选股公式、固定池阈值、调度时间已经非常接近；剩余大部分目标差异来自弱市状态不一致。

## 弱市边界根因

本轮弱市边界日志定位到一个明确问题：同花顺多个边界日缺 `000510.SH` 指数数据。

典型例子：`2020-02-11`

- 同花顺：`000510.SH:NA`
- 聚宽：`000510.XSHG:2020-02-10:4445.8474:4434.9985:above`

这会直接改变状态机判断：

- 同花顺只看到 `below=1, above=2`，继续保持弱市。
- 聚宽看到 `below=1, above=3`，满足退出弱市条件，转为非弱市。

同样的问题也出现在 `2020-04-29`、`2020-04-30`、`2020-07-23`、`2020-08-18`、`2020-08-19`、`2021-02-09`、`2021-06-24`、`2021-08-03` 等边界日。

这解释了为什么弱市不一致长期存在：不是 MA 公式本身错，而是同花顺指数源对 `000510` 缺失或代码映射失败。

## 阈值口径

V3 固定池阈值已经对齐：

- 全周期阈值 ratio 中位数：`0.9932`
- 最小：`0.9447`
- 最大：`1.0272`

年度中位数也稳定在 0.99 附近：

- 2020：`1.0001`
- 2021：`0.9891`
- 2022：`0.9881`
- 2023：`0.9903`
- 2024：`0.9956`
- 2025：`0.9968`
- 2026：`0.9950`

这说明 V3 的固定池阈值是可比口径。后续如果重新打开全市场阈值，就要单独解决同花顺全市场 ETF 元数据不足问题。

## 执行层差异

信号接近后，执行层问题更突出：

- 同花顺：`1047` 条 WARN，主要是下单股数为 0、资金不足、部分未成交。
- 聚宽：`626` 条 ERROR/WARNING，其中委托错误 `626` 条；订单目标金额中位数约 `363.18 万`，最大约 `9483.99 万`。

这些错误不会改变日志里的目标信号，但会改变实际资金曲线。因此下一阶段必须拆分：

1. 信号一致性。
2. 可成交一致性。
3. 平台资金曲线一致性。

## 合理解释

V3 做到了两件事：

1. 把同花顺速度和日志体积压下来。
2. 把固定池成交额阈值修到和聚宽几乎一致。

但 V3 没有提升弱市匹配率，因为真正的问题是 `000510` 指数数据缺失。只要同花顺缺这个指数，很多边界日会少一个 above/below 票数，弱市进入/退出都会偏离聚宽。

因此，当前最合理的下一步不是继续改动态池，也不是调评分参数，而是修 `000510` 指数源。

## 下一轮迭代建议

### 1. 修复同花顺 `000510` 指数源

优先级最高。

建议在同花顺脚本里为 `000510` 增加多代码候选：

- `000510.SH`
- `000510.XSHG`
- `000510`
- 如平台支持，再尝试指数专用后缀或别名。

并加单独日志：

```text
WUFU_INDEX_RESOLVE code=000510.SH resolved=... rows=... last_date=...
```

目标：边界日不再出现 `000510.SH:NA`。

### 2. 复跑固定池 V4

保持：

- `USE_DYNAMIC_POOL = False`
- `USE_FULL_MARKET_THRESHOLD = False`

只修指数源，不改其它逻辑。

目标：

- 弱市匹配率从 `89.84%` 提升到 `95%+`
- 目标匹配率从 `90.60%` 提升到 `95%+`

### 3. 统一执行层

弱市修完后，再处理：

- 100 份最小交易单位。
- 买入目标金额向下取整。
- 卖出不足 100 份时两边统一忽略或保留。
- 资金不足时两边统一为“最大可买份额”。

### 4. 最后再恢复全市场动态池

只有固定池信号和执行层都稳定后，再打开：

- `USE_FULL_MARKET_THRESHOLD = True`
- `USE_DYNAMIC_POOL = True`

否则全市场 ETF 元数据问题会再次混入，难以判断是动态池错还是弱市错。
