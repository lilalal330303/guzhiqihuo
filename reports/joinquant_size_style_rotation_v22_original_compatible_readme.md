# 大小盘反复横跳 V2.2 原始兼容版：回测与对照说明

策略脚本：[joinquant_size_style_rotation_v22_original_compatible.py](./joinquant_size_style_rotation_v22_original_compatible.py)

## 1. 版本定位

V2.2 的主路径是“原始行为兼容 + 必要安全修复”，不是在 V2.1 上继续调参。V2.1 将原策略的成分股横截面信号替换为指数价格收益、趋势确认、波动率标准化、滞回和 risk-off 空仓逻辑，改变了原始策略的信号来源与持仓行为；用户提供的回测对照显示，这条路径相对原版出现了实质性退化，因此不再作为主路径。

用户此前提供的原始回测截图仅作为参考观察，不是本地可复现的绩效指标。本文和本地测试均不声称已经得到任何聚宽云端回测收益，也不据截图反推或填写绩效数字。

## 2. 原始兼容信号

V2.2 使用国证 2000（`399303.XSHE`）和中证 500（`399905.XSHE`）在信号日可见的历史成分股：

1. 对每个指数读取截至 `context.previous_date` 的最近 20 个日频收盘价；
2. 分别计算每只可用成分股的首尾收益，再取横截面平均值，得到 `mean_2000` 和 `mean_500`；
3. 使用 `mean_2000 / mean_500 > 1.2` 判断分支；
4. 条件成立时进入原始命名的 `BIG` 分支，否则进入原始命名的 `SMALL` 分支。

`BIG`/`SMALL` 的名称与指数直觉存在反向感，但这是原始代码的有效分支方向。V2.2 有意保留该命名和方向，不做“语义纠正”，以免再次改变基线行为。分母无效、成分池为空、价格数据不可用或有效样本不足时，本次信号返回不可用并跳过调仓，现有持仓保持不变。

## 3. B0 默认行为

B0 保留原始策略的主要运行约定：每月第一个交易日 09:30 调仓、目标持仓 5 只、可用现金在缺失目标间等额分配、`SMALL` 分支最高价格 10 元、默认零滑点，并保留最近涨停持仓的保护处理。09:05 准备持仓状态，14:00 检查昨日涨停持仓是否开板。

脚本顶部的 `DEFAULT_PARAMS` 是聚宽运行时的默认配置，`initialize` 会复制它到 `g.params`：

```python
DEFAULT_PARAMS = {
    "stock_num": 5,
    "style_window": 20,
    "ratio_threshold": 1.2,
    "min_style_samples": 2,
    "max_price": 10.0,
    "min_listing_days": 375,
    "recent_limit_days": 40,
    "winsorize_returns": False,
    "market_guard": False,
    "market_guard_ma": 60,
    "slippage": 0.0,
    "use_historical_constituents": True,
    "big_use_filtered_pool": False,
}
```

其中 `big_use_filtered_pool=False` 是刻意的 B0 兼容选择：原始 `BIG` 分支实际把未经后续过滤的全市场 `stocks` 传入基本面查询，而不是把过滤后的 `choice` 传入。改为 `True` 可能更符合一般直觉，但会改变候选池，必须作为独立后续实验，不能混入 B0 基线。

## 4. 安全修复的边界

这些改动用于消除信息时点、数据形状和下单失败的不确定性，不用于创造新的择时信号：

- 历史日期：指数成分调用显式使用 `date=context.previous_date` 对应的截止日；所有基本面查询显式使用 `date=context.previous_date`；风格价格序列以该日期为 `end_date`。信号不读取调仓日收盘后数据。
- 数据形状：统一处理聚宽可能返回的长表、宽表、单列、MultiIndex 和 Panel-like 收盘价数据；对缺失值、非数值、无穷值、零价格、空池和样本不足做显式处理。
- 调仓安全：信号或目标为空时不清仓；卖出非目标持仓后，仅对缺失目标分配可用现金；下单前检查停牌和涨跌停状态，并处理异常及空订单。
- 顺序安全：候选排名去重，仍在候选中的原持仓优先保留，受保护的昨日涨停持仓计入目标数量，避免重复目标和意外超配。

本版本在月度调仓事件中生成信号并通过 `order_target_value` 执行，保持原始月度流程；安全修复不代表本地环境可以复现聚宽撮合结果。

## 5. B0–B4 单变量实验矩阵

每组实验都应从 B0 开始，只改变表中列出的变量。除 B3/B4 明确比较成本外，其余组保持 `slippage=0.0`；除 B1 外保持 `winsorize_returns=False`；除 B2 外保持 `market_guard=False`。

| 组别 | 相对 B0 的设置 | 目的与比较方式 |
| --- | --- | --- |
| B0 原始兼容基线 | 使用全部 `DEFAULT_PARAMS`，包括 `big_use_filtered_pool=False` | 首先与原始策略在同一聚宽回测设置下对照，确认信号方向、交易频率和持仓行为是否恢复 |
| B1 收益截尾 | 仅设 `winsorize_returns=True` | 对成分股横截面收益做 5%/95% 截尾，检查极端个股是否主导风格信号 |
| B2 市场保护 | 仅设 `market_guard=True` | 使用 `000985.XSHG` 最近 60 日收盘均线保护，比较回撤改善与收益损失；保护未通过时保持现有持仓 |
| B3 滑点敏感性 | 分别设 `slippage=0`、`0.001`、`0.002` | 其余参数固定为 B0，观察撮合成本对结果的敏感程度 |
| B4 时段与成本对照 | 在各比较窗口内重复 B0，并结合 B3 的成本档位 | 检查结论是否只依赖单一市场阶段或零滑点假设；不在该组同时引入其他信号改动 |

## 6. 回测窗口与指标

所有组别至少使用以下四个窗口，并保证聚宽中的初始资金、基准、复权、佣金、税费和调度设置一致：

- 全样本：`2020-01-01` 至 `2026-07-18`；
- 早期样本：`2020-01-01` 至 `2021-12-31`；
- 中期样本：`2022-01-01` 至 `2023-12-31`；
- 近期样本：`2024-01-01` 至 `2026-07-18`。

每次运行至少记录：总收益、年化收益、最大回撤、交易次数、胜率、换手率、空目标次数，以及滑点敏感性。空目标次数应按“当次应调仓但候选或最终目标为空、因此保持现有持仓”的统一口径统计。滑点敏感性应并列展示 `0 / 0.001 / 0.002` 三档结果及相对 B0 零滑点结果的变化，不能只报告最优档位。

建议同时保存参数快照、按月交易清单和空目标日志，便于区分信号差异、候选池差异与撮合失败。

## 7. 聚宽验证顺序

1. 将脚本完整复制到聚宽 Python 3 策略环境，先不修改 `DEFAULT_PARAMS`，运行 B0。
2. 在完全相同的云端回测设置下，把 B0 与原始策略逐期比较；优先核对分支、目标持仓、调仓次数和空目标日志，再比较绩效。
3. B0 行为确认后，按 B1、B2、B3、B4 顺序做单变量实验，并分别保存四个时间窗口的结果。
4. 只有当某项改动在全样本和各分段均未显著破坏 B0 时，才考虑把它纳入后续候选版本。

本地测试只验证纯函数、默认参数、数据形状兼容、目标构造和历史日期静态约束；它没有调用聚宽云端数据与撮合引擎。因此，必须由用户在聚宽中实际运行脚本，并首先完成 B0 与原始版本的同口径比较。
