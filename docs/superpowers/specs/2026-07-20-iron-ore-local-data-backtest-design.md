# 铁矿石 CTA 本地数据与回测环境设计规格

## 目标

为 V1.6 铁矿石 CTA 建立一套可重复的“聚宽研究端导出 → 本地 DuckDB 入库 → 本地日频回测”流程。导出的数据必须能够支持主连信号、实际交易合约价格、近月合约选择、到期换月和回测质量审计。

## 数据口径

### 必须下载

1. **主连日线 `I8888.XDCE`**：`trade_date, open, high, low, close`，用于 V1.6 的均线、坡度、效率比、方向一致性、ATR 和波动率计算；同时保留 `volume, money, open_interest` 作为质量诊断。
2. **实际铁矿石合约日线 `I####.XDCE`**：同样的 OHLCV、成交额和持仓量，至少覆盖回测区间前的预热区间。实际合约用于下一交易日开盘成交、持仓估值和换月，不能只下载主连。
3. **合约元数据**：`symbol, list_date, end_date`，用于排除信号日期尚未上市或距离到期不足 8 天的合约。
4. **点时合约宇宙**：`asof_date, symbol, list_date, end_date`，由聚宽研究端按交易日导出，避免本地使用最终时点的合约列表产生未来合约泄漏。

### 不作为 V1.6 首版必需

- 分钟线不是首版必需，因为 V1.6 的信号只读取日线，首版本地回测以“信号日 T、下一交易日 T+1 开盘执行”的透明日频近似为主。
- 结算价、交割库、基差和宏观数据不参与当前 V1.6 信号，不在首版下载范围内。

## 文件与数据库设计

聚宽研究端导出五个文件：

- `iron_ore_main_daily.csv`
- `iron_ore_contract_daily.csv`
- `iron_ore_contracts.csv`
- `iron_ore_universe_daily.csv`
- `manifest.json`

本地导入到 `data/market.duckdb` 的四张铁矿石专用表：

- `iron_ore_main_daily`
- `iron_ore_contract_daily`
- `iron_ore_contracts`
- `iron_ore_universe_daily`

日线表沿用本地标准字段，并额外保留 `open_interest`、`source`、`fetched_at`。所有主键和重复行在导入时检查；OHLC 必须为正，日期必须有序且不能晚于导出截止日。

## 本地回测口径

- 每个信号日只能读取主连截至当日的历史数据。
- 交易目标合约使用当日点时合约宇宙，过滤 `list_date <= signal_date` 且 `end_date - signal_date > 8` 天后选择最早到期合约。
- 信号在下一交易日开盘执行，价格使用实际合约日线开盘价并加固定滑点；不使用信号日之后的数据确定信号。
- 期货权益按合约乘数 100、保证金率 15%、手续费和最大保证金使用率计算；允许多空对称持仓。
- 本地结果必须输出信号、成交、每日权益和基础指标，并明确这是日频执行近似，不承诺逐笔复现聚宽分钟回测。

## 质量门槛

- 主连日线覆盖回测起始日前至少 80 个交易日。
- 主连日期唯一、OHLC 无空值和非正值。
- 合约日线主键 `(symbol, trade_date)` 唯一，所有合约代码匹配 `^I\d{4}\.XDCE$`。
- 每个目标交易日能找到点时可用合约；若找不到，回测必须记录为 `no_contract`，不能静默选择未来合约。
- 质量报告输出行数、日期范围、合约数、重复数、缺失字段和点时宇宙覆盖率。
